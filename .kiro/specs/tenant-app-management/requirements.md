# Requirements Document

## Introduction

This feature introduces per-tenant app management and app-scoped push MFA. Each tenant can create multiple named apps, each identified by a unique `appId` and protected by a generated secret. The push MFA endpoint is extended to require an `appId` and authenticate callers via an app secret (Bearer token). Tenant Admins manage their apps through a new "Apps" tab in the Tenant Admin Dashboard. Every tenant is provisioned with a non-deletable default app at creation time (and retroactively for existing tenants). The default app is also used by Tenant Admins to simulate push notifications to their users directly from the dashboard.

## Glossary

- **System**: The full-stack application (ASP.NET Core backend + React/TypeScript frontend).
- **Backend**: The ASP.NET Core API.
- **Web_App**: The React/TypeScript frontend.
- **Tenant**: An isolated organisational unit, as defined in the existing multi-tenant model.
- **Tenant_Admin**: A user with the `TenantAdmin` role who manages apps and users within their tenant.
- **App**: A named entity belonging to a tenant, identified by a UUID `appId` and authenticated by a 32-character alphanumeric secret.
- **App_Secret**: A cryptographically random 32-character alphanumeric string generated by the Backend and used to authenticate push MFA requests on behalf of an App.
- **Default_App**: A special App created automatically for each tenant that cannot be deleted. Used by the Tenant_Admin to simulate push notifications.
- **Push_MFA_Endpoint**: The `POST /push` endpoint that initiates a push MFA challenge for a user.
- **App_Management_API**: The backend endpoints under `/tenant/apps` for CRUD operations on Apps.
- **Tenant_Admin_Dashboard**: The frontend page accessible to Tenant Admins, containing "Users" and "Apps" tabs.
- **Simulate_Push**: An action available to the Tenant_Admin in the dashboard to send a test push notification to a selected user using the Default_App.

## Requirements

### Requirement 1: App Data Model

**User Story:** As a developer, I want a well-defined data model for tenant apps, so that the system can persist and query app records reliably.

#### Acceptance Criteria

1. THE Backend SHALL define a `TenantApps` table with columns: `Id` (UUID, PK), `TenantId` (UUID, FK → Tenants), `Name` (varchar), `Secret` (varchar), `IsDisabled` (boolean, default false), `IsDefault` (boolean, default false), `CreatedAt` (timestamptz).
2. THE Backend SHALL enforce a unique constraint on `(TenantId, Name)` so app names are unique per tenant.
3. THE Backend SHALL apply all schema changes via EF Core migrations.
4. THE Backend SHALL add a navigation property from `Tenant` to its collection of `TenantApp` entities.

---

### Requirement 2: Default App Provisioning

**User Story:** As a system operator, I want every tenant to have a default app automatically, so that push MFA works out of the box and existing tenants are not broken.

#### Acceptance Criteria

1. WHEN a new tenant is created, THE Backend SHALL automatically create a Default_App for that tenant with `IsDefault = true` and a freshly generated App_Secret.
2. THE Backend SHALL apply a data migration that creates a Default_App for every existing tenant that does not already have one.
3. THE Default_App SHALL have its `Name` set to `"Default"`.

---

### Requirement 3: App Management API

**User Story:** As a Tenant Admin, I want to create, list, edit, disable, and delete apps in my tenant, so that I can control which external systems can trigger push MFA.

#### Acceptance Criteria

1. THE App_Management_API SHALL expose `GET /tenant/apps` that returns all Apps belonging to the authenticated Tenant Admin's tenant, including `id`, `name`, `isDefault`, `isDisabled`, and `createdAt` (the App_Secret SHALL NOT be included in list responses).
2. THE App_Management_API SHALL expose `POST /tenant/apps` that creates a new App with a provided `name` and a freshly generated App_Secret, returning the full App record including the App_Secret (only time it is returned in full).
3. WHEN a `POST /tenant/apps` request is received with a name that already exists within the same tenant, THE App_Management_API SHALL return HTTP 409.
4. THE App_Management_API SHALL expose `PATCH /tenant/apps/{appId}` that updates the `name` and/or `isDisabled` fields of the specified App.
5. WHEN a `PATCH /tenant/apps/{appId}` request targets an App belonging to a different tenant, THE App_Management_API SHALL return HTTP 403.
6. THE App_Management_API SHALL expose `DELETE /tenant/apps/{appId}` that removes the specified App.
7. WHEN a `DELETE /tenant/apps/{appId}` request targets the Default_App (`IsDefault = true`), THE App_Management_API SHALL return HTTP 409 with error `cannot_delete_default_app`.
8. WHEN a `DELETE /tenant/apps/{appId}` request targets an App belonging to a different tenant, THE App_Management_API SHALL return HTTP 403.
9. THE App_Management_API SHALL expose `POST /tenant/apps/{appId}/reset-secret` that generates and stores a new App_Secret for the specified App, returning the new secret.
10. WHEN a `POST /tenant/apps/{appId}/reset-secret` request targets an App belonging to a different tenant, THE App_Management_API SHALL return HTTP 403.
11. WHEN any `/tenant/apps` request is received without a valid Tenant Admin JWT, THE App_Management_API SHALL return HTTP 401.

---

### Requirement 4: App Secret Generation

**User Story:** As a developer, I want app secrets to be strong and randomly generated, so that they cannot be guessed or brute-forced.

#### Acceptance Criteria

1. THE Backend SHALL generate App_Secrets using a cryptographically secure random number generator.
2. THE Backend SHALL generate App_Secrets as exactly 32 alphanumeric characters (A–Z, a–z, 0–9).
3. THE Backend SHALL store App_Secrets as plain text in the database (they are opaque tokens, not passwords).

---

### Requirement 5: App-Authenticated Push MFA Endpoint

**User Story:** As an external caller, I want to trigger a push MFA challenge by providing a tenantId, username, appId, and app secret, so that the system can authenticate the caller and route the challenge to the correct user.

#### Acceptance Criteria

1. WHEN a `POST /push` request is received, THE Push_MFA_Endpoint SHALL require the `Authorization: Bearer <app-secret>` header, `tenantId`, `username`, and `appId` fields in the request body.
2. WHEN the `Authorization` header is absent or malformed, THE Push_MFA_Endpoint SHALL return HTTP 401.
3. WHEN the provided `appId` does not exist within the specified `tenantId`, THE Push_MFA_Endpoint SHALL return HTTP 401.
4. WHEN the Bearer token does not match the App_Secret of the specified App, THE Push_MFA_Endpoint SHALL return HTTP 401.
5. WHEN the specified App has `IsDisabled = true`, THE Push_MFA_Endpoint SHALL return HTTP 403 with error `app_disabled`.
6. WHEN the specified `username` does not exist within the specified `tenantId`, THE Push_MFA_Endpoint SHALL return HTTP 404 with error `user_not_found`.
7. WHEN no push subscription is registered for the resolved user, THE Push_MFA_Endpoint SHALL return HTTP 404 with error `device_not_found`.
8. WHEN all validations pass, THE Push_MFA_Endpoint SHALL dispatch a Web Push notification to the user's registered device and initiate the long-poll response flow as per the existing push MFA behaviour.
9. THE Push_MFA_Endpoint SHALL no longer accept the legacy `userId` field; callers MUST provide `tenantId`, `username`, and `appId`.

---

### Requirement 6: Apps Tab in Tenant Admin Dashboard

**User Story:** As a Tenant Admin, I want an "Apps" tab in my dashboard, so that I can manage my tenant's apps without using raw API calls.

#### Acceptance Criteria

1. THE Tenant_Admin_Dashboard SHALL display two tabs: "Users" (existing functionality) and "Apps" (new).
2. WHEN the "Apps" tab is active, THE Web_App SHALL display a list of all Apps for the tenant, showing `name`, `isDefault`, `isDisabled`, and `createdAt`.
3. THE Web_App SHALL provide a form to create a new App by entering a name; upon successful creation THE Web_App SHALL display the generated App_Secret once in a clearly highlighted, copyable field, with a warning that it will not be shown again.
4. THE Web_App SHALL provide a button to reset an App's secret; upon success THE Web_App SHALL display the new App_Secret once in a clearly highlighted, copyable field.
5. THE Web_App SHALL provide a button to enable or disable an App (toggle `isDisabled`), with the Default_App's disable toggle available.
6. THE Web_App SHALL provide a button to delete an App; THE Web_App SHALL display a confirmation prompt before deletion.
7. WHEN a delete is attempted on the Default_App, THE Web_App SHALL display an error message indicating the default app cannot be deleted.
8. WHEN any App management action fails, THE Web_App SHALL display a human-readable error message.

---

### Requirement 7: Simulate Push Notification (Tenant Admin)

**User Story:** As a Tenant Admin, I want to send a test push notification to a user from the dashboard, so that I can verify that a user's device is correctly registered.

#### Acceptance Criteria

1. THE Tenant_Admin_Dashboard SHALL provide a "Simulate Push" action for each user in the Users tab.
2. WHEN the Tenant Admin triggers "Simulate Push" for a user, THE Web_App SHALL call the Push_MFA_Endpoint using the Default_App's `appId` and App_Secret.
3. WHEN the push is dispatched successfully, THE Web_App SHALL display the result (accepted / denied / timed out) to the Tenant Admin.
4. WHEN the target user has no registered device, THE Web_App SHALL display a human-readable error indicating the user has no device registered.
5. IF the Default_App is disabled, THEN THE Web_App SHALL display an error indicating the default app is disabled and simulation is unavailable.
